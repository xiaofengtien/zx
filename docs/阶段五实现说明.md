# 阶段五实现说明

## 一、设计原则确认

### ✅ 优先方案：ZIP包方案（性能最优）

**核心思路**：
- 后端生成ZIP包，包含所有数据（试卷、题目、媒体文件）
- 客户端下载ZIP包，解压后使用
- **一个ZIP包 = 所有数据**，最大程度优化客户端性能

**优势**：
- ✅ 一次下载，包含所有数据
- ✅ 减少网络请求次数
- ✅ 减少数据库操作
- ✅ 支持离线使用

### ⚠️ 备选方案：业务数据表同步（兼容方案）

**使用场景**：
- 后端还没有生成ZIP包
- 或者ZIP包生成失败
- 临时兼容方案

**劣势**：
- ❌ 需要多次API调用
- ❌ 需要同步多个表
- ❌ 性能较差

---

## 二、当前实现状态

### ✅ 已实现（阶段五）

1. **表结构创建**：
   - `paper_package` - 存储ZIP包（BLOB或文件路径）
   - `paper_index` - 试卷索引
   - `question_media_index` - 媒体文件索引
   - 业务数据表（paper、question等）- 备选方案使用

2. **ZIP包检查逻辑**：
   - `shouldSyncBusinessData()` - 检查是否有ZIP包
   - 如果有ZIP包 → 跳过业务数据同步
   - 如果没有ZIP包 → 同步业务数据表

3. **业务数据同步**（备选方案）：
   - 同步试卷、题目、分类等表数据
   - 仅在无ZIP包时使用

### ❌ 未实现（阶段六）

1. **ZIP包下载功能**：
   - `downloadPaperPackage(paperCode)` - 下载ZIP包
   - 支持断点续传（HTTP Range）
   - 支持分片下载（>200MB）

2. **ZIP包解压功能**：
   - `extractPaperPackage(paperCode)` - 解压ZIP包
   - 提取媒体文件到本地
   - 保存媒体文件索引

3. **ZIP包同步逻辑**：
   - 检查版本号（package_hash）
   - 增量同步
   - 全量同步

## 三、正确的数据流

### 理想流程（ZIP包方案）

```
1. 后端生成ZIP包
   ↓
2. 客户端检查是否有ZIP包
   ↓
3. 如果没有 → 下载ZIP包
   ↓
4. 解压ZIP包
   ↓
5. 提取媒体文件到本地
   ↓
6. 保存索引到SQLite
   ↓
7. 使用本地数据答题
```

### 当前流程（备选方案）

```
1. 客户端检查是否有ZIP包
   ↓
2. 如果没有 → 同步业务数据表
   ↓
3. 使用SQLite数据答题
```

## 四、需要提醒的差异点

### ⚠️ 重要提醒

1. **当前实现是备选方案**：
   - 阶段五实现了业务数据表同步
   - 这是为了兼容没有ZIP包的情况
   - **不是主要方案**

2. **ZIP包功能还未实现**：
   - 下载ZIP包（阶段六）
   - 解压ZIP包（阶段六）
   - 媒体文件提取（阶段六）

3. **性能差异**：
   - ZIP包方案：1次下载，包含所有数据
   - 表同步方案：多次API调用，性能较差

4. **建议**：
   - 如果后端已经可以生成ZIP包，应该优先实现阶段六
   - 业务数据表同步只是临时兼容方案
