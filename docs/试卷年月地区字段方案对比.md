# 试卷年月地区字段方案对比

## 需求确认

- **月份格式**：具体月份（1-12），需要字典
- **省份编码**：显示简称（如"河北"），需要字典
- **自定义名称**：非必填
- **唯一约束**：年月 + 省份 + 试卷类型
- **分组条件**：年月 + 省份 + 试卷类型（用于paper-package分组）
- **试卷名称格式**：`{year}年{month}月{province}{paper_type_label}{custom}`

## 方案对比

### 方案1：最小改动（利用paper_code）

#### 改动清单

**数据库（MySQL）**：
- ✅ 无需新增字段
- ✅ 无需修改表结构
- ✅ 无需修改唯一约束（paper_code已有UNIQUE）

**数据库（SQLite）**：
- ✅ 无需新增字段
- ✅ 无需修改表结构

**后端代码**：
- 📝 修改 `PaperBizImpl.generatePaperCode()` - 生成逻辑（~20行）
- 📝 修改 `PaperServiceImpl.createPaper()` - 验证和生成paper_code（~10行）
- 📝 修改 `PaperServiceImpl.updatePaper()` - 验证唯一性（~10行）
- 📝 修改 `PaperServiceImpl.generatePaperPackage()` - 分组路径（~5行）
- 📝 新增工具类：解析paper_code的工具方法（~30行）
- 📝 修改 `Paper.java` - 无需修改（字段不变）
- 📝 修改 `PaperDTO.java` - 无需修改
- 📝 修改 `PaperCreateBO.java` - 新增4个字段（year, month, province, customName）（~10行）
- 📝 修改 `PaperUpdateBO.java` - 新增4个字段（~10行）
- 📝 修改 `PaperPageBO.java` - 新增查询字段（可选，~5行）

**前端代码**：
- 📝 修改 `add.vue` - 调整表单布局，新增4个字段（~50行）
- 📝 修改 `edit.vue` - 调整表单布局，新增4个字段（~50行）
- 📝 修改 `index.vue` - 列表显示（可选，~20行）
- 📝 新增字典数据：月份字典、省份字典

**客户端代码（SQLite）**：
- ✅ 无需修改（paper_code字段已存在）

**总计**：~220行代码修改 + 2个字典DML

#### 优点
- ✅ 改动最小
- ✅ 无需数据库迁移
- ✅ 向后兼容（现有paper_code仍可用）
- ✅ 唯一约束自动满足

#### 缺点
- ⚠️ paper_code变长（但仍在VARCHAR(100)范围内）
- ⚠️ 需要解析paper_code获取年月、地区（但可以缓存）
- ⚠️ 如果后续规则变化（如只使用年+类型），需要重新生成paper_code

#### 扩展性分析
- **扩展场景1**：只使用年+类型（去掉月份、省份）
  - 需要：修改paper_code生成逻辑
  - 需要：重新生成所有paper_code（数据迁移）
  - 影响：中等

- **扩展场景2**：增加季度字段
  - 需要：修改paper_code生成逻辑
  - 需要：重新生成所有paper_code（数据迁移）
  - 影响：中等

**总计**：~300行代码修改 + 数据库迁移 + 2个字典DML

#### 优点
- ✅ 字段清晰，查询方便

- ✅ 符合数据库设计规范

#### 缺点

- ⚠️ 需要数据迁移（现有数据需要补全year、month、province）
- ⚠️ SQLite不支持复合唯一索引，需要在应用层验证

#### 扩展性分析

  - 需要：字段可以保留，只是不使用
  - 影响：小

- **扩展场景2**：增加季度字段

  - 需要：修改唯一约束
  - 影响：中等

## 推荐方案

### 推荐：方案1（最小改动）

**理由**：
1. ✅ **改动最小**：只需修改paper_code生成逻辑，无需数据库迁移
2. ✅ **风险最低**：不修改表结构，不影响现有数据
3. ✅ **实现最快**：可以快速上线
4. ✅ **向后兼容**：现有paper_code仍可用

**后续扩展处理**：
- 如果规则变化，可以：
  1. 修改paper_code生成逻辑（新数据使用新规则）
  2. 旧数据可以保留（向后兼容）
  3. 或者通过数据迁移脚本统一更新

### 备选：方案2（如果后续需要频繁扩展）

**适用场景**：
- 如果预期后续会频繁修改规则
- 如果需要频繁按年月、省份查询
- 如果团队更倾向于规范的数据库设计
