# 媒体文件存储架构设计

## 一、问题分析

### 1.1 用户疑问

**问题**：客户端需要本地存储媒体文件，那么后台上传时是否也需要存到本地？

**分析**：
- 后台（服务端）：管理员上传题目时上传媒体文件
- 客户端（离线端）：需要离线访问媒体文件
- 关键问题：后台和客户端的存储策略如何统一？

---

## 二、架构设计

### 2.1 整体架构

```
┌─────────────────────────────────────────────────────────────┐
│                     后台服务端（MySQL）                        │
├─────────────────────────────────────────────────────────────┤
│  1. 管理员上传题目 + 媒体文件                                  │
│     ↓                                                         │
│  2. 媒体文件存储：OSS（或服务器本地文件系统）                    │
│     ↓                                                         │
│  3. 数据库记录：question_media 表（存储 OSS URL/路径）          │
│     ↓                                                         │
│  4. 生成试卷包：将题目+媒体文件打包成 ZIP                        │
│     ↓                                                         │
│  5. ZIP包存储：OSS（或服务器本地文件系统）                       │
└─────────────────────────────────────────────────────────────┘
                            ↓ 同步
┌─────────────────────────────────────────────────────────────┐
│                   客户端（SQLite + 本地文件）                   │
├─────────────────────────────────────────────────────────────┤
│  1. 下载试卷包（ZIP）                                          │
│     ↓                                                         │
│  2. 解压试卷包                                                │
│     ↓                                                         │
│  3. 提取媒体文件 → 存储到本地 media/ 目录                       │
│     ↓                                                         │
│  4. 数据库记录：question_media_index 表（存储本地相对路径）     │
│     ↓                                                         │
│  5. 答题时从本地路径加载媒体文件                                │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 存储策略对比

| 存储位置 | 用途 | 访问方式 | 优缺点 |
|---------|------|---------|--------|
| **后台：OSS/服务器** | 原始文件存储 | HTTP/HTTPS URL | ✅ 统一管理<br>✅ 在线访问<br>❌ 离线无法访问 |
| **后台：ZIP包** | 试卷包存储 | HTTP/HTTPS 下载 | ✅ 批量下载<br>✅ 包含所有媒体文件 |
| **客户端：本地文件** | 离线访问 | 本地文件路径 | ✅ 离线可用<br>✅ 访问速度快<br>❌ 占用磁盘空间 |

## 三、推荐方案：分层存储架构

### 3.1 后台存储策略

#### 方案A：OSS存储（推荐，适合生产环境）

**存储位置**：
- 媒体文件：OSS（如阿里云OSS、腾讯云COS）
- ZIP包：OSS

**优点**：
- ✅ 高可用、高并发
- ✅ CDN加速
- ✅ 自动备份
- ✅ 节省服务器存储空间

**缺点**：
- ⚠️ 需要OSS服务费用
- ⚠️ 需要配置OSS访问权限

**实现**：
```java
// 后台上传媒体文件
@Service
public class MediaUploadService {
    
    @Autowired
    private OssUtil ossUtil; // OSS工具类
    
    /**
     * 上传媒体文件到OSS
     */
    public String uploadMediaFile(MultipartFile file, String paperCode, Integer questionId) {
        // 1. 生成OSS路径
        String ossPath = String.format("media/%s/q_%d/%s", 
            paperCode, questionId, file.getOriginalFilename());
        
        // 2. 上传到OSS
        String ossUrl = ossUtil.upload(file, ossPath);
        
        // 3. 保存到数据库
        QuestionMedia media = new QuestionMedia();
        media.setQuestionId(questionId);
        media.setMediaName(file.getOriginalFilename());
        media.setMediaUrl(ossUrl); // OSS URL
        media.setMediaPath(ossPath); // OSS路径
        media.setMediaSize(file.getSize());
        media.setStorageType(0); // 0-OSS存储
        questionMediaMapper.insert(media);
        
        return ossUrl;
    }
}
```

#### 方案B：服务器本地存储（适合开发/小规模）

**存储位置**：
- 媒体文件：服务器本地文件系统（如 `/data/media/`）
- ZIP包：服务器本地文件系统

**优点**：
- ✅ 无需OSS费用
- ✅ 实现简单
- ✅ 适合内网环境

**缺点**：
- ⚠️ 需要管理服务器存储空间
- ⚠️ 需要备份策略
- ⚠️ 单点故障风险

**实现**：
```java
// 后台上传媒体文件
@Service
public class MediaUploadService {
    
    @Value("${media.upload.path:/data/media}")
    private String mediaUploadPath;
    
    /**
     * 上传媒体文件到服务器本地
     */
    public String uploadMediaFile(MultipartFile file, String paperCode, Integer questionId) {
        // 1. 生成本地路径
        String relativePath = String.format("media/%s/q_%d/%s", 
            paperCode, questionId, file.getOriginalFilename());
        String localPath = Paths.get(mediaUploadPath, relativePath).toString();
        
        // 2. 保存到本地
        File localFile = new File(localPath);
        localFile.getParentFile().mkdirs();
        file.transferTo(localFile);
        
        // 3. 生成访问URL（通过Nginx等）
        String accessUrl = String.format("/media/%s", relativePath);
        
        // 4. 保存到数据库
        QuestionMedia media = new QuestionMedia();
        media.setQuestionId(questionId);
        media.setMediaName(file.getOriginalFilename());
        media.setMediaUrl(accessUrl); // 访问URL
        media.setMediaPath(localPath); // 本地路径
        media.setMediaSize(file.getSize());
        media.setStorageType(2); // 2-本地文件系统
        questionMediaMapper.insert(media);
        
        return accessUrl;
    }
}
```

### 3.2 客户端存储策略

**存储位置**：
- ZIP包：`paper_packages/{paper_code}_v{version}.zip`
- 媒体文件：`media/{paper_code}/q_{question_id}/`

**实现**：
```javascript
// 客户端同步试卷包
class PaperService {
  /**
   * 同步试卷包并解压媒体文件
   */
  async syncPaperPackage(paperCode) {
    // 1. 从服务端下载ZIP包
    const zipData = await this.downloadPaperPackage(paperCode)
    
    // 2. 保存ZIP包到本地
    const zipPath = path.join(this.packageBasePath, `${paperCode}_v${version}.zip`)
    fs.writeFileSync(zipPath, zipData)
    
    // 3. 解压ZIP包
    const tempDir = path.join(this.appDataPath, 'temp', paperCode)
    await this.extractZip(zipData, tempDir)
    
    // 4. 提取媒体文件到本地
    const mediaDir = path.join(this.mediaBasePath, paperCode)
    await this.extractMediaFiles(tempDir, mediaDir)
    
    // 5. 保存索引到SQLite
    await this.saveMediaIndex(paperCode, mediaDir)
  }
}
```

## 四、数据流转流程

### 4.1 完整流程

```
┌─────────────────────────────────────────────────────────────┐

│  3. 移动媒体文件到本地：                                       │

                            ↓
```

## 五、数据库设计

### 5.1 后台（MySQL）：question_media 表

```sql
CREATE TABLE `question_media` (
  `id` INT PRIMARY KEY AUTO_INCREMENT,
  `question_id` INT NOT NULL COMMENT '题目ID',
  `media_type` TINYINT NOT NULL COMMENT '媒体类型：1-题目媒体，2-选项媒体',
  `option_id` INT COMMENT '选项ID',
  `media_name` VARCHAR(255) NOT NULL COMMENT '媒体文件名',
  `media_path` VARCHAR(500) COMMENT '服务器路径（OSS路径或本地路径）',
  `media_url` VARCHAR(500) COMMENT '访问URL（OSS URL或HTTP URL）',
  `media_size` INT COMMENT '文件大小（字节）',
  `media_format` VARCHAR(50) COMMENT '文件格式',
  `storage_type` TINYINT DEFAULT 0 COMMENT '存储类型：0-OSS，2-本地文件系统',
  `create_time` DATETIME DEFAULT CURRENT_TIMESTAMP,
  INDEX `idx_question_id` (`question_id`)
) COMMENT='题目媒体文件表（后台）';
```

### 5.2 客户端（SQLite）：question_media_index 表

```sql
CREATE TABLE `question_media_index` (
  `id` INTEGER PRIMARY KEY AUTOINCREMENT,
  `paper_id` INTEGER NOT NULL,
  `question_id` INTEGER NOT NULL,
  `media_type` INTEGER NOT NULL COMMENT '1-题目媒体，2-选项媒体',
  `option_id` INTEGER,
  `media_name` TEXT NOT NULL,
  `media_path` TEXT NOT NULL COMMENT '本地相对路径',
  `media_format` TEXT,
  `media_size` INTEGER,
  INDEX `idx_paper_question` (`paper_id`, `question_id`)
) COMMENT='题目媒体文件索引表（客户端）';
```

## 六、关键问题解答

### 6.1 后台上传时是否需要存到本地？

**答案**：**不需要存到客户端本地，但需要存到服务器本地或OSS**

**原因**：
1. **后台存储**：用于在线访问和生成试卷包
   - 存储到OSS（推荐）或服务器本地文件系统
   - 客户端通过HTTP/HTTPS访问（在线时）
   
2. **客户端存储**：用于离线访问
   - 从试卷包（ZIP）解压后存储到本地
   - 客户端通过本地文件路径访问（离线时）

3. **数据流转**：
   ```
   后台上传 → OSS/服务器 → 生成ZIP包 → 客户端下载 → 解压到本地
   ```

### 6.2 为什么需要两层存储？

**原因**：
- **后台存储**：统一管理，支持在线访问，生成试卷包
- **客户端存储**：支持离线访问，提高访问速度

**类比**：
- 就像手机APP下载视频：服务器存储原始视频，APP下载后缓存到本地

### 6.3 如何保证数据一致性？

**策略**：
1. **版本管理**：ZIP包包含版本号，客户端检查版本
2. **哈希校验**：ZIP包包含SHA256哈希值，客户端校验
3. **增量同步**：客户端检查 `package_hash`，不一致时重新下载

## 七、推荐方案总结

### 7.1 后台存储

✅ **推荐：OSS存储**（生产环境）
- 媒体文件上传到OSS
- ZIP包上传到OSS
- 数据库记录OSS URL/路径

⚠️ **备选：服务器本地存储**（开发/小规模）
- 媒体文件保存到服务器本地
- ZIP包保存到服务器本地
- 通过Nginx等提供HTTP访问

### 7.2 客户端存储

✅ **必须：本地文件存储**
- 从ZIP包解压媒体文件到本地
- SQLite存储本地相对路径
- 答题时从本地路径加载

### 7.3 数据流转

```
后台上传 → OSS/服务器 → 生成ZIP → 客户端下载 → 解压到本地
```
