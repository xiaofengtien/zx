# 异步生成试卷包方案设计

## 一、问题分析

当前 `/paper/generatePackage` 接口是同步的，生成ZIP包需要较长时间（查询数据、生成ZIP、上传OSS），容易导致请求超时。

## 二、方案设计

### 方案A：基于内存的任务管理（推荐，简单快速）

#### 后端实现

1. **任务管理服务**
   - 使用 `ConcurrentHashMap` 存储任务状态（内存存储）
   - 任务ID：使用 `paperId` 作为唯一标识
   - 任务状态：`PENDING`（等待中）、`RUNNING`（执行中）、`SUCCESS`（成功）、`FAILED`（失败）
   - 任务进度：0-100 的整数百分比
   - 任务结果：成功时存储版本号，失败时存储错误信息

2. **异步执行**
   - 使用 `@Async` 注解或 `ThreadPoolTaskExecutor` 线程池
   - 在异步任务中更新进度和状态

3. **API接口**
   - `POST /paper/generatePackage` - 提交生成任务（立即返回任务ID）
   - `GET /paper/getPackageTaskStatus?paperId={paperId}` - 查询任务状态和进度

#### 前端实现

1. **提交任务**
   - 调用 `/paper/generatePackage` 接口
   - 立即返回，显示"任务已提交，正在生成..."

2. **轮询查询进度**
   - 使用 `setInterval` 每 2-3 秒轮询一次 `/paper/getPackageTaskStatus`
   - 显示进度条（0-100%）
   - 显示当前步骤（如："正在查询数据..."、"正在生成ZIP..."、"正在上传OSS..."）

3. **完成处理**
   - 进度达到 100% 或状态为 `SUCCESS` 时，停止轮询
   - 显示成功提示
   - 状态为 `FAILED` 时，显示错误信息

#### 优点
- 实现简单，无需引入额外依赖
- 响应快速，用户体验好
- 适合单机部署

#### 缺点
- 任务状态存储在内存中，服务重启会丢失
- 不适合集群部署（多实例间无法共享状态）

---

### 方案B：基于Redis的任务管理（推荐用于生产环境）

#### 后端实现

1. **任务管理服务**
   - 使用 Redis 存储任务状态（key: `paper:package:task:{paperId}`）
   - 任务信息包含：状态、进度、开始时间、完成时间、错误信息
   - 设置过期时间（如：1小时）

2. **异步执行**
   - 使用 `@Async` 注解或线程池
   - 在异步任务中更新 Redis 中的进度和状态

3. **API接口**
   - 同方案A

#### 优点
- 支持集群部署
- 任务状态持久化（Redis）
- 可以设置过期时间自动清理

#### 缺点
- 需要Redis依赖
- 实现稍复杂

---

### 方案C：基于数据库的任务管理

#### 后端实现

1. **任务表设计**
   ```sql
   CREATE TABLE `paper_package_task` (
     `id` INT PRIMARY KEY AUTO_INCREMENT,
     `paper_id` INT NOT NULL,
     `task_status` VARCHAR(20) NOT NULL, -- PENDING/RUNNING/SUCCESS/FAILED
     `progress` INT DEFAULT 0, -- 0-100
     `current_step` VARCHAR(200), -- 当前步骤描述
     `error_message` TEXT, -- 错误信息
     `create_time` DATETIME,
     `update_time` DATETIME,
     `finish_time` DATETIME
   );
   ```

2. **异步执行**
   - 创建任务记录（状态：PENDING）
   - 异步执行，更新任务状态和进度

3. **API接口**
   - 同方案A

#### 优点
- 任务状态持久化
- 可以查询历史任务
- 支持集群部署

#### 缺点
- 需要创建新表
- 需要定期清理历史任务

---

## 三、推荐方案

**推荐使用方案A（内存管理）+ 方案B（Redis）的混合方案：**

1. **开发/测试环境**：使用方案A（简单快速）
2. **生产环境**：使用方案B（支持集群）

通过配置开关切换两种实现。

## 四、详细设计（方案A + 方案B混合）

### 4.1 后端设计

#### 任务状态枚举
```java
public enum PackageTaskStatus {
    PENDING,   // 等待中
    RUNNING,   // 执行中
    SUCCESS,   // 成功
    FAILED     // 失败
}
```

#### 任务信息类
```java
public class PackageTaskInfo {
    private Integer paperId;
    private PackageTaskStatus status;
    private Integer progress;  // 0-100
    private String currentStep; // 当前步骤描述
    private String errorMessage; // 错误信息
    private Long startTime;
    private Long finishTime;
    private Integer newVersion; // 生成的新版本号
}
```

#### 接口设计

**1. 提交生成任务**
```
POST /paper/generatePackage
Request: { "id": 1 }
Response: { 
    "code": 200,
    "msg": "任务已提交",
    "data": {
        "paperId": 1,
        "taskId": "paper_1_1234567890"
    }
}
```

**2. 查询任务状态**
```
GET /paper/getPackageTaskStatus?paperId=1
Response: {
    "code": 200,
    "data": {
        "paperId": 1,
        "status": "RUNNING", // PENDING/RUNNING/SUCCESS/FAILED
        "progress": 65,      // 0-100
        "currentStep": "正在上传ZIP包到OSS...",
        "errorMessage": null,
        "newVersion": null   // 成功时返回新版本号
    }
}
```

#### 进度节点设计

生成ZIP包的步骤和进度分配：
1. **查询数据**（0-20%）
   - 查询试卷信息：5%
   - 查询卷别列表：5%
   - 查询大题列表：5%
   - 查询题目列表：5%

2. **生成ZIP**（20-70%）
   - 生成manifest.json：10%
   - 生成questions.json：10%
   - 下载并打包媒体文件：30%（根据媒体文件数量动态分配）

3. **上传OSS**（70-90%）
   - 上传ZIP包：20%

4. **完成**（90-100%）
   - 更新数据库：5%
   - 验证完成：5%

### 4.2 前端设计

#### UI组件

1. **进度条对话框**
   - 使用 Element UI 的 `el-dialog` + `el-progress`
   - 显示进度条（0-100%）
   - 显示当前步骤文字
   - 显示"取消"按钮（可选）

2. **调用流程**
   ```
   用户点击"生成试卷包"按钮
   ↓
   调用 generatePackage API（立即返回）
   ↓
   显示进度对话框
   ↓
   开始轮询 getPackageTaskStatus（每2秒一次）
   ↓
   更新进度条和步骤文字
   ↓
   进度达到100%或状态为SUCCESS/FAILED
   ↓
   停止轮询，显示结果
   ```

#### 代码示例（伪代码）

```javascript
// 提交任务
async handleGeneratePackage(paperId) {
  try {
    const response = await generatePaperPackage({ id: paperId })
    // 显示进度对话框
    this.showProgressDialog = true
    this.progress = 0
    this.currentStep = '任务已提交，等待处理...'
    
    // 开始轮询
    this.startPolling(paperId)
  } catch (error) {
    this.$message.error('提交任务失败：' + error.message)
  }
}

// 轮询查询进度
startPolling(paperId) {
  this.pollingTimer = setInterval(async () => {
    try {
      const response = await getPackageTaskStatus(paperId)
      const taskInfo = response.data
      
      this.progress = taskInfo.progress
      this.currentStep = taskInfo.currentStep
      
      if (taskInfo.status === 'SUCCESS') {
        clearInterval(this.pollingTimer)
        this.$message.success(`试卷包生成成功，版本：v${taskInfo.newVersion}`)
        this.showProgressDialog = false
        // 刷新列表
        this.loadPaperList()
      } else if (taskInfo.status === 'FAILED') {
        clearInterval(this.pollingTimer)
        this.$message.error('生成失败：' + taskInfo.errorMessage)
        this.showProgressDialog = false
      }
    } catch (error) {
      console.error('查询进度失败：', error)
    }
  }, 2000) // 每2秒查询一次
}
```

## 六、注意事项

1. **任务去重**：同一试卷同时只能有一个生成任务
2. **超时处理**：如果任务执行超过30分钟，自动标记为失败
3. **错误处理**：捕获异常，更新任务状态为FAILED
4. **资源清理**：任务完成后，延迟清理任务信息（如：5分钟后）
5. **并发控制**：使用锁机制防止同一试卷重复提交任务
