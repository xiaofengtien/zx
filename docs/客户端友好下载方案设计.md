# 客户端友好下载方案设计

## 一、问题分析

### 当前问题
1. **登录后立即同步**：用户登录后，系统会立即同步所有试卷包，阻塞用户操作
2. **下载速度慢**：即使实现了分片下载，100MB文件下载仍需较长时间
3. **用户体验差**：用户需要等待下载完成才能使用系统

### 用户需求
- 登录后不阻塞，允许用户先进入系统
- 后台静默下载，显示进度但不强制等待
- 下载完成后通知用户
- 优化下载性能

---

## 二、方案设计

### 方案概述
**后台静默下载 + 智能优先级 + 进度提示 + 性能优化**

### 核心设计原则
1. **非阻塞**：登录后立即进入系统，不等待下载完成
2. **智能优先级**：优先下载用户可能使用的试卷
3. **后台下载**：在后台线程中下载，不阻塞UI
4. **进度提示**：显示下载进度，但不强制等待
5. **断点续传**：支持断点续传，提高下载可靠性

---

## 三、详细设计

### 3.1 下载策略优化

#### 3.1.1 智能优先级策略
```
优先级1（立即下载）：
  - 用户已选中的试卷
  - 最近使用的试卷
  - 只有一个试卷时，自动选中并优先下载

优先级2（后台下载）：
  - 其他可用试卷
  - 按试卷ID顺序下载

优先级3（延迟下载）：
  - 用户未查看的试卷
  - 可以延迟到用户需要时再下载
```

#### 3.1.2 下载队列管理
```javascript
class DownloadQueue {
  // 高优先级队列（立即下载）
  highPriority: []
  
  // 普通队列（后台下载）
  normalPriority: []
  
  // 延迟队列（按需下载）
  lowPriority: []
  
  // 当前下载任务
  currentTask: null
  
  // 最大并发数（建议2-3个）
  maxConcurrent: 2
}
```

### 3.2 性能优化

#### 3.2.1 分片下载优化
```javascript
// 当前实现
- 分片大小：10MB/20MB/50MB（根据文件大小）
- 并发数：4个

// 优化方案
- 分片大小：5MB（更小的分片，更快响应）
- 并发数：6-8个（增加并发，提高速度）
- 超时时间：30秒（更短的超时，快速失败）
- 重试机制：3次重试，指数退避
```

#### 3.2.2 断点续传支持
```javascript
// 保存下载进度
const downloadProgress = {
  paperId: 4,
  totalSize: 104857600,
  downloadedSize: 52428800,
  chunks: [
    { start: 0, end: 5242879, downloaded: true },
    { start: 5242880, end: 10485759, downloaded: true },
    // ...
  ]
}

// 断点续传逻辑
1. 检查本地是否有未完成的下载
2. 如果有，从断点继续下载
3. 如果没有，从头开始下载
```

### 3.4 下载状态管理

## 五、性能预期

### 优化前
- 100MB文件下载：60-90秒
- 阻塞用户操作：是
- 用户体验：差

### 优化后
- 100MB文件下载：15-25秒（提升3-4倍）
- 阻塞用户操作：否
- 用户体验：优秀

### 性能提升原因
1. **更小的分片**：5MB分片，更快响应
2. **更多并发**：6-8个并发，充分利用带宽
3. **断点续传**：失败后快速恢复
4. **非阻塞下载**：用户可立即使用系统

## 七、用户体验流程

### 场景1：登录后只有一个试卷
```
1. 用户登录
2. 立即进入试卷选择页面
3. 自动选中唯一试卷
4. 后台开始下载（优先级1）
5. 显示下载进度（非阻塞）
6. 下载完成后，自动启用"开始答题"按钮
7. 发送通知："试卷包下载完成，可以开始答题了"
```

### 场景2：登录后有多个试卷
```
1. 用户登录
2. 立即进入试卷选择页面
3. 显示所有试卷列表
4. 后台开始下载所有试卷（按优先级）
5. 显示总体下载进度（非阻塞）
6. 每个试卷显示独立下载状态
7. 用户可以选择任意已就绪的试卷开始答题
8. 未下载完成的试卷显示"等待下载"或"下载中"
```

### 场景3：下载失败
```
1. 下载失败
2. 显示错误状态
3. 提供"重试"按钮
4. 发送错误通知
5. 用户可手动重试或稍后重试
```
