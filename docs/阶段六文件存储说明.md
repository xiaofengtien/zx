# 阶段六：文件存储说明

## 一、文件存储位置

### 1. ZIP包存储

#### 小文件（<50MB）
- **存储方式**：SQLite BLOB
- **存储位置**：`paper_package` 表的 `package_data` 字段
- **文件系统**：**不会在文件系统中看到ZIP文件**
- **原因**：直接存储在数据库中，提高小文件的访问速度

#### 中等文件（50-200MB）
- **存储方式**：文件系统
- **存储位置**：`{userData}/paper_packages/{paper_code}_v{version}.zip`
- **文件系统**：可以在文件系统中看到ZIP文件

#### 大文件（>200MB）
- **存储方式**：文件系统（分片下载）
- **存储位置**：`{userData}/paper_packages/{paper_code}_v{version}.zip`
- **状态**：功能待实现

### 2. 解压后的文件

#### 临时文件（temp目录）
- **存储位置**：`{userData}/temp/{paper_code}/`
- **内容**：
  - `manifest.json` - 试卷元数据
  - `questions.json` - 题目数据
  - `media/` - 媒体文件（临时）
- **清理时机**：解压完成后立即删除（第594行：`fs.rmSync(tempDir, { recursive: true, force: true })`）
- **原因**：临时文件，解压后提取媒体文件到media目录，然后删除

#### 媒体文件（media目录）
- **存储位置**：`{userData}/media/{paper_code}/`
- **目录结构**：
  ```
  media/
  └── {paper_code}/
      ├── intro/
      │   └── {intro_audio_file}  # 试卷独白音频
      └── q_{question_id}/
          ├── {title_media_file}  # 题目媒体文件
          └── options/
              └── {option_media_file}  # 选项媒体文件
  ```
- **持久化**：这些文件会一直保留，直到清理旧版本

### 3. 数据库存储

#### paper_package表
- **字段**：
  - `package_data` (BLOB) - 小文件的ZIP包数据（<50MB）
  - `package_path` (TEXT) - 大文件的ZIP包路径（>=50MB）
  - `storage_type` (INTEGER) - 0=BLOB, 1=文件系统
  - `package_hash` (TEXT) - ZIP包的SHA256哈希值
  - `package_size` (INTEGER) - ZIP包大小（字节）

#### question_media_index表
- **字段**：
  - `paper_id` - 试卷ID
  - `question_id` - 题目ID
  - `media_type` - 媒体类型（intro/title/option）
  - `media_name` - 媒体文件名
  - `media_path` - 媒体文件相对路径（相对于media目录）

## 二、为什么看不到ZIP文件和解压的JSON文件？

### 1. ZIP文件（小文件情况）
- **原因**：如果ZIP包小于50MB，会存储在SQLite的BLOB字段中，不会在文件系统中创建ZIP文件
- **验证方法**：
  ```sql
  SELECT paper_code, storage_type, package_size, 
         CASE WHEN storage_type = 0 THEN 'BLOB存储' ELSE '文件系统存储' END as storage_info
  FROM paper_package;
  ```
- **查看BLOB数据**：
  ```sql
  SELECT paper_code, LENGTH(package_data) as blob_size
  FROM paper_package
  WHERE storage_type = 0;
  ```

### 2. 解压的JSON文件（manifest.json和questions.json）
- **原因**：这些文件存储在temp目录中，解压完成后会被立即删除
- **代码位置**：`paperService.js` 第594行
  ```javascript
  // 6. 清理临时目录
  fs.rmSync(tempDir, { recursive: true, force: true })
  ```
- **原因**：JSON数据已经解析并存储到SQLite中，不需要保留临时文件

### 3. 媒体文件
- **位置**：`{userData}/media/{paper_code}/`
- **应该能看到**：如果解压成功，媒体文件应该在这个目录下

## 四、Hash验证说明

### 1. Hash验证的作用
- **数据完整性**：确保下载的ZIP包没有被损坏或篡改
- **版本一致性**：确保下载的ZIP包与服务器上的版本一致

### 2. Hash不匹配的原因
1. **网络传输错误**：下载过程中数据损坏
2. **服务器文件更新**：服务器上的ZIP包已更新，但数据库中的hash未更新
3. **下载内容错误**：下载的不是ZIP文件，而是错误信息

### 3. 当前问题分析
从日志看：
- **期望hash**：`f9c72f49bfbc56847bcba63ee8db93356d0359d32d715b50797f055a8899bebd`
- **实际hash**：`fed64545c9f9dde1a830ebbebf6bdd09a68d849b74f7c10212e2baa9f85de411`
- **文件大小**：961字节（太小，正常的ZIP包应该更大）

**可能的原因**：
1. 后端下载的ZIP包内容不正确（只有961字节，可能是错误响应）
2. 数据库中的hash是旧版本的hash
3. 需要重新生成试卷包并更新数据库中的hash

## 五、建议的解决方案

### 1. 检查后端下载的ZIP包
- 查看后端日志，确认从OSS下载的ZIP包大小和hash
- 如果后端下载的ZIP包hash与数据库中的hash不一致，需要重新生成试卷包

### 2. 重新生成试卷包
1. 在后台管理-试卷管理中，重新生成试卷包
2. 确保数据库中的`package_hash`和`package_size`已更新
3. 客户端重新同步

### 3. 验证ZIP包内容
- 检查后端下载的ZIP包是否真的是ZIP文件（前2字节应该是0x50 0x4B，即"PK"）
- 如果后端下载的内容不是ZIP文件，需要检查OSS上的文件是否正确
