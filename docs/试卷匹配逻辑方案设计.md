# 试卷匹配逻辑方案设计

## 一、问题分析

### 当前情况
1. **试卷编码格式**：`{year}{month}_{province_code}_{paper_type}_{custom_name}`
   - 示例：`202312_hebei_junior_high_school_english_listening_模拟一`
   - 包含：**年月** + **省份** + **试卷类型** + **自定义名称**

2. **学员档案配置**：
   - `applicable_papers` 字段：存储试卷类型（paper_type）列表
   - 例如：`["junior_high_school_english_listening", "high_school_english_listening"]`
   - **只配置了试卷类型，没有配置年月和省份**

3. **问题**：
   - 如果只根据试卷类型查询，会返回一批试卷（不同年月、省份、自定义名称）
   - 用户不想在客户端让学员选择试卷
   - 需要自动匹配到唯一试卷

## 二、解决方案对比

### 方案A：在学员档案中增加精确配置字段（推荐）

#### 方案描述
在学员档案中增加字段，精确配置学员适用的试卷：
- `applicable_paper_codes`：存储具体的试卷编码列表（paper_code）
- 或者：`applicable_paper_configs`：存储配置对象（包含年月、省份、类型）

#### 数据结构
```sql
-- 方案A1：直接存储试卷编码列表
ALTER TABLE student_archive 
ADD COLUMN applicable_paper_codes TEXT COMMENT '适用试卷编码列表（JSON数组，如：["202312_hebei_junior_high_school_english_listening_模拟一"]）';

-- 方案A2：存储配置对象（更灵活）
ALTER TABLE student_archive 
ADD COLUMN applicable_paper_configs TEXT COMMENT '适用试卷配置（JSON数组，格式：[{"year":2023,"month":12,"province":"hebei","paperType":"junior_high_school_english_listening","customName":"模拟一"}]）';
```

#### 优点
- ✅ **精确匹配**：直接指定试卷编码，不会有多条结果
- ✅ **灵活配置**：可以为不同学员配置不同试卷
- ✅ **客户端简单**：直接根据配置查询，无需选择

#### 缺点
- ⚠️ **需要修改数据库**：增加字段
- ⚠️ **配置复杂**：需要管理员精确配置每个学员的试卷

#### 适用场景
- 学员数量不多，可以精确配置
- 需要为不同学员分配不同试卷

---

### 方案B：自动匹配规则（根据学员信息自动选择）

#### 方案描述
根据学员档案中的信息（省份、当前年月等），自动匹配试卷：
1. 从学员档案获取：省份、当前年月（或配置的年月）
2. 根据试卷类型 + 省份 + 年月查询试卷
3. 如果有多条，选择最新版本或指定规则

#### 数据结构
```sql
-- 在学员档案中增加字段
ALTER TABLE student_archive 
ADD COLUMN province VARCHAR(50) COMMENT '省份编码（字典值：paper_province）';

ALTER TABLE student_archive 
ADD COLUMN exam_year INTEGER COMMENT '考试年份（用于匹配试卷）';

ALTER TABLE student_archive 
ADD COLUMN exam_month INTEGER COMMENT '考试月份（用于匹配试卷）';
```

#### 匹配规则
```javascript
// 客户端匹配逻辑
const paperType = studentArchive.applicablePapers[0] // 试卷类型
const province = studentArchive.province              // 省份
const year = studentArchive.examYear || new Date().getFullYear()  // 年份
const month = studentArchive.examMonth || new Date().getMonth() + 1 // 月份

// 查询试卷：paper_type + province + year + month
const papers = await queryPapers({
  paperType,
  province,
  year,
  month
})

// 如果有多条，选择最新版本或第一条
const selectedPaper = papers[0] // 或按版本号排序取最新
```

#### 优点
- ✅ **自动化**：无需精确配置每个学员的试卷编码
- ✅ **灵活**：可以根据学员信息自动匹配

#### 缺点
- ⚠️ **需要学员档案有省份信息**：可能需要补充数据
- ⚠️ **匹配规则复杂**：如果有多条结果，需要明确选择规则
- ⚠️ **可能匹配不到**：如果该年月省份没有对应试卷

#### 适用场景
- 学员档案已有省份信息
- 试卷按年月省份规律分布
- 可以接受自动匹配规则

### 方案C：服务端智能匹配接口（推荐）

#### 方案描述
客户端调用服务端接口，服务端根据学员信息自动匹配并返回唯一试卷：
1. 客户端传递：学员ID + 试卷类型
2. 服务端根据学员档案信息（省份、年月等）自动匹配
3. 返回唯一试卷编码或试卷信息

#### 接口设计
```java
// 服务端接口
@PostMapping("/student/paper/getMatchedPaper")
public Result getMatchedPaper(@RequestBody MatchedPaperBO bo) {
    // 1. 获取学员档案
    StudentArchive archive = getStudentArchive(bo.getAppUserId());
    
    // 2. 根据学员信息匹配试卷
    // - 从 applicable_papers 获取试卷类型
    // - 从 archive.province 获取省份
    // - 从 archive.examYear/examMonth 或当前年月获取
    // - 查询匹配的试卷
    
    // 3. 如果有多条，按规则选择（最新版本、最新创建时间等）
    
    // 4. 返回唯一试卷
    return Result.success(matchedPaper);
}
```

#### 客户端调用
```javascript
// 客户端调用
const matchedPaper = await ipcRenderer.invoke('paper:getMatchedPaper', {
  paperType: selectedPaperType,
  appUserId: userInfo.userId
})

// 返回：{ paperCode, paperId, paperName, ... }
```

#### 优点
- ✅ **客户端简单**：只需调用接口，无需复杂匹配逻辑
- ✅ **服务端统一**：匹配规则在服务端统一管理
- ✅ **灵活扩展**：可以增加更复杂的匹配规则

#### 缺点
- ⚠️ **需要网络**：必须在线才能获取
- ⚠️ **需要服务端开发**：增加接口

#### 适用场景
- 需要复杂的匹配规则
- 希望匹配逻辑在服务端统一管理
- 可以接受在线匹配

### 方案D：混合方案（推荐）

#### 方案描述
结合方案A和方案C：
1. **优先使用精确配置**：如果学员档案有 `applicable_paper_codes`，直接使用
2. **降级到自动匹配**：如果没有精确配置，调用服务端接口自动匹配
3. **客户端缓存**：匹配结果缓存到本地，支持离线使用

#### 数据结构
```sql
-- 增加精确配置字段（可选）
ALTER TABLE student_archive 
ADD COLUMN applicable_paper_codes TEXT COMMENT '适用试卷编码列表（JSON数组，优先级最高）';

-- 增加匹配辅助字段（可选）
ALTER TABLE student_archive 
ADD COLUMN province VARCHAR(50) COMMENT '省份编码（用于自动匹配）';
ALTER TABLE student_archive 
ADD COLUMN exam_year INTEGER COMMENT '考试年份（用于自动匹配）';
ALTER TABLE student_archive 
ADD COLUMN exam_month INTEGER COMMENT '考试月份（用于自动匹配）';
```

#### 匹配流程
```javascript
// 客户端匹配逻辑
async function getMatchedPaper(paperType, appUserId) {
  // 1. 获取学员档案
  const archive = await getStudentArchive(appUserId)
  
  // 2. 优先使用精确配置
  if (archive.applicablePaperCodes && archive.applicablePaperCodes.length > 0) {
    // 根据试卷类型找到对应的试卷编码
    const paperCode = findPaperCodeByType(archive.applicablePaperCodes, paperType)
    if (paperCode) {
      return { paperCode, source: 'exact_config' }
    }
  }
  
  // 3. 降级到服务端自动匹配
  const matchedPaper = await callServerMatchAPI(paperType, archive)
  return { ...matchedPaper, source: 'auto_match' }
}
```

#### 优点
- ✅ **灵活**：支持精确配置和自动匹配
- ✅ **兼容**：可以逐步迁移到精确配置
- ✅ **可靠**：有降级方案

#### 缺点
- ⚠️ **实现复杂**：需要支持两种匹配方式

## 三、推荐方案

### 推荐：方案D（混合方案）

#### 理由
1. **灵活性高**：支持精确配置和自动匹配
2. **渐进式迁移**：可以先使用自动匹配，后续逐步精确配置
3. **可靠性好**：有降级方案

#### 实施步骤

**第一步：数据库调整**

## 四、确认问题

### 1. 学员档案当前有哪些字段？
- 是否有 `province`（省份）字段？
- 是否有 `hometown`（籍贯）字段？
- 是否有其他可以用于匹配的字段？

### 2. 试卷匹配规则
- **年月**：使用当前年月，还是学员档案中配置的年月？
- **省份**：使用学员档案中的省份，还是其他规则？
- **多条结果**：如果匹配到多条试卷，选择规则是什么？
  - 最新版本？
  - 最新创建时间？
  - 指定排序规则？

### 3. 配置方式偏好
- **精确配置**：是否愿意为每个学员精确配置试卷编码？
- **自动匹配**：是否希望系统自动匹配？
- **混合方式**：是否接受混合方案？
